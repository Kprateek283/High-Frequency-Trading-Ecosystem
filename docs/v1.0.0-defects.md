# Known Defects in v1.0.0 — and how to fix them

Ten defects found by exercising the tagged `v1.0.0` build. Each entry is written
to be actionable without prior knowledge of the project: what is wrong, how to
see it for yourself, where it lives, what to change, and how to know you fixed
it. No code is given deliberately — the fix descriptions state intent and
constraints so the implementer can write it in the surrounding style.

These are **defects** (something is incorrect), not enhancements. Deferred
improvements and accepted limitations are listed separately at the end so nobody
mistakes them for work items.

> ## Status: all ten are FIXED on `main` (unreleased)
>
> The code defects were fixed first, deliberately: fixing them changes what the
> documentation must say, so doing the docs first would have made them stale
> twice. Each fix was reproduced before and verified after — the measurements are
> in the commit messages.
>
> | ID | Fix | Evidence |
> | :-- | :--- | :--- |
> | D2 | anchor re-published every ~100ms, rate re-derived over full uptime | heartbeat drift −1.291ms/27s → −0.023ms; 30/30 samples positive over 90s |
> | D3 | undefined fill ratio replaced with exactly-derivable trade count | panel reads 10,000 against liquidity's known 10,000 matches |
> | D4 | crossed-book repair + TTL eviction for stranded orders | phantom surviving 5,000 clean cycles now cleared |
> | D5 | replay deals orders round-robin over N connections | 0 matches → 1,165,314 on the same 5M-order file |
> | D1, D6–D10 | documentation reconciled with the code | claims re-checked against a live run |
>
> The entries below are kept as written, because the reasoning is the useful part
> and the reproductions still work as regression checks. A future release should
> archive this file the way `known-issues.md` was archived.

## Before you start

Everything below assumes a working build. From the repository root:

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
cd build && ctest && cd ..          # C++ suite
python3 -m pip install -r requirements.txt
python3 -m monitoring.run_tests     # Python suite
```

Binaries land in `build/bin/`. The benchmark entry point is
`./scripts/run_sharding.sh` (run it from the repo root). Throughput is measured
by `python3 scripts/measure_throughput.py`.

Two rules that apply to every item here:

- **Locate code by grep or symbol name, never by the line numbers quoted in this
  document.** They drift the moment anyone edits a file. Line numbers appear
  below only to make a claim checkable on the tagged commit.
- **Do not commit a red test suite.** Both suites are green at `v1.0.0`; any
  failure you see after a change is yours.

## Suggested order

Work the phases in order. Within a phase the items are independent.

| Phase | Items | Why this order |
| :-- | :-- | :--- |
| **A — Documentation accuracy** | D1, D6, D7, D8, D9, D10 | No build risk, no behaviour change, and it stops the shipped docs actively misinforming readers. Biggest credibility win per unit effort. |
| **B — Monitoring only** | D4 | Python-only, isolated to `monitoring/`, cannot affect the engine. |
| **C — Engine-touching** | D2, D3, D5 | Require a rebuild, and D3 changes a shared-memory layout that the Python side mirrors. Do these last and one at a time. |

# Phase A — Documentation accuracy

## D1. `docs/dependency.md` describes a system that no longer exists

**Severity:** high (misinforms every reader) · **Effort:** medium · **Risk:** none

### What is wrong

`dependency.md` is a capability and interface map written *before* the
implementation plan was executed. It was never updated afterwards. It now ships
as a current-state document while describing the pre-implementation world.

Its interface status table marks:

| Interface | Doc says | Reality in v1.0.0 |
| :-- | :-- | :-- |
| I3 (TCP gateway) | "transport ready, **symbol contract broken**" | Fixed. One canonical `STK#####` scheme. |
| I4 (`/dev/shm/hft_stats`) | "✖ **needs §4**" | Implemented. Readable by `monitoring/feeds/stats_reader.py`. |
| I5 (TSC anchor) | "✖ **needs §3**" | Implemented, inside the stats region. |
| I6 (`orders.bin`) | "✅ readable; ✖ **contents useless**" | Contents are valid; symbols decode. |
| I8 (launch barrier) | "✖ **needs §7**" | Implemented. Used by `run_sharding.sh` and `orchestrator.py`. |
| I7 (control socket) | "✖ not built, not specified" | **Still accurate.** Leave it. |

The prose is wrong in the same direction. The "I6 is not pcap" block states the
file's symbols are `AAPL/MSFT/GOOG/AMZN`, that the gateway decodes them to `999`,
and that "**Every order in it is rejected.**" The "I3's blocker (review A1)"
block tells the reader not to expect fills.

### How to see it

```bash
./build/bin/generate_pcap /tmp/orders.bin     # then replay it against the engine
```

The generated file uses `encode_symbol` with instruments 0–3, i.e. `STK00000`–
`STK00003`. A 5,000,000-order replay run yields **1,999,996 accepted (NEW)**, not
zero. Compare against the doc's claim that every order is rejected.

### The fix

Bring the document to current state, keeping it a *map* rather than turning it
into a changelog:

1. Update the I1–I8 status column. Only I7 remains unbuilt.
2. Rewrite the "I6 is not pcap" block. The genuinely useful part — that the file
   is a flat `OuchEnterOrder[]` with no pcap framing, so no Python pcap library
   is needed — is still true and should stay. Delete the symbol/rejection claims.
3. Delete the "I3's blocker (review A1)" block, or reduce it to one sentence
   noting the constraint was resolved. Do not silently drop the fact that a
   symbol contract exists — state what it is.
4. Sweep the Python tier tables for capability cells that assume the blocked
   state (search the file for `A1`, `blocked`, `✖`).
5. Add a line near the top stating what the document is and when it was last
   reconciled with the code, so the next reader can judge its freshness.

### Verify

Search the file for `needs §`, `A1`, `blocked`, `useless`, `999`. Every surviving
hit must describe something still true. Cross-check each remaining `✖` against a
real run.

## D6. `messages.h` contradicts its own encoder in a comment

**Severity:** trivial · **Effort:** trivial · **Risk:** none

The `OuchEnterOrder` struct annotates the symbol field as
`char stock[8]; // e.g. "AAPL    "` (around line 99), sitting directly beneath
the `encode_symbol`/`decode_symbol` pair that makes exactly that encoding

**Fix:** change the example to a canonical symbol (`STK00000`). While in the

**Verify:** no symbol example in `hft_engine/src/protocol/messages.h` fails
`decode_symbol`.

# Phase B — Monitoring only

## D4. The reconstructed order book never prunes, so drift is permanent

**Severity:** medium (grows with uptime) · **Effort:** medium · **Risk:** low

### What is wrong

`_BookFeed` in `monitoring/tui/app.py` owns a `BookSet` that is created once and
never reset or expired. `monitoring/core/orderbook.py` removes an order only when
a matching execute or cancel message arrives for it.

Market data is UDP with no sequence numbers, so some loss is unavoidable (see
*Accepted limitations*). Every lost execute/cancel strands its order in the book
**forever**. The consequences compound with uptime:

- memory grows without bound in a dashboard intended to run continuously;
- top-of-book degrades monotonically, because a stranded order at a better price
  than the live market permanently shadows the true best bid or ask.

A short run looks fine, which is why this passes casual testing.

### How to see it

Run the engine, run `./build/bin/liquidity` (10,000 crossing pairs that end with
an empty book), and watch the TUI's top-of-book panel. The instrument should
finish flat. Occasionally it retains resting orders and shows a stale best ask
with no bid. Repeat runs against one long-lived TUI process and the count
accumulates — it never returns to zero on its own.

### The fix

Pick one; the first is the smaller change, the second is the more correct one.

**Option 1 — bound the staleness.** Track when each resting order was last
touched and drop orders untouched for longer than a configurable interval. Also
drop price levels once they hold no orders. This bounds memory and lets a
shadowed book heal, at the cost of evicting genuinely long-resting orders. Choose
the interval deliberately and name it as a constant with a comment explaining the
trade-off.

**Option 2 — periodic resynchronisation.** Rebuild the book from scratch on an
interval, or when a heuristic suggests drift (for example the book has crossed —
best bid at or above best ask — which the engine's matching guarantees can never
happen in reality, so it is a reliable corruption signal). This is self-healing
and does not evict valid orders, but discards good state on every rebuild.

Whichever you choose, a crossed book is worth surfacing in the health panel
rather than hiding: it means the consumer's view is provably wrong.

### Verify

Add a self-test to the module's `_selftest` covering the mechanism you chose: for
option 1, that a stale order is evicted and a fresh one is not; for option 2,
that a deliberately corrupted book returns to a sane state. Then run the live
check: drive repeated `liquidity` runs against a single long-lived dashboard
process and confirm the resting-order count returns to zero rather than climbing.

`python3 -m monitoring.run_tests` must stay green.

# Phase C — Engine-touching

Do these one at a time, rebuilding and running both suites after each.

## D2. The TSC anchor never recalibrates, so heartbeat age can go negative

**Severity:** medium · **Effort:** medium · **Risk:** medium (touches timing)

### What is wrong

`Timer::calibrate()` in `hft_engine/src/core/timer.h` derives `cycles_per_ns`
once at startup from a single 50 ms sample, and nothing refreshes it. That value
is published in the stats region as part of the TSC anchor and is what Python
uses to convert engine timestamps to wall-clock time.

Any error in that one sample is a *rate* error, so the divergence between
converted time and real time grows without bound the longer the engine runs.

It is already observable within seconds: a measured heartbeat age of **-0.27 ms**
— the monitor concluding the engine's heartbeat arrived from the future.

The second half of the defect is that nothing catches it. `monitoring/health.py`
tests only whether the age exceeds a staleness ceiling. A negative age passes
every check, so the liveness signal is silently unreliable in one direction.

### How to see it

Start the engine, then read the heartbeat age through
`monitoring/orchestrator.py`'s `heartbeat_age_ns()` or the TUI's liveness line.
Negative values appear readily. Leave the engine running and the magnitude of the
error grows.

### The fix

Two independent parts. **Do both** — either alone leaves a hole.

**Engine side.** Refresh the anchor periodically rather than trusting one startup
sample. The natural home is the thread that already updates the stats region on
an interval; re-reading a wall clock alongside the TSC and republishing the
anchor is cheap and off the hot path. Constraints: the anchor is published inside
a seqlock-protected region, so update it the same way as the other fields so
readers cannot observe a torn anchor; and this is a monitoring path, so keep it
out of the order-matching hot path entirely.

**Monitor side.** Treat a negative age as the anomaly it is. A small negative
value is normal clock jitter and should be tolerated; a large one means the
anchor is wrong and should be reported, not silently accepted. Decide the
threshold deliberately and document it next to the existing staleness ceiling.

### Verify

Add a health self-test asserting that a large negative age produces a warning
rather than an OK verdict. For the engine side, run for several minutes and
confirm the heartbeat age stays small and non-negative throughout — the current
build's error grows visibly over that span.

## D3. `fill_ratio` divides two counters that do not measure the same thing

**Severity:** medium (shows a plausible but wrong number) · **Effort:** medium

### What is wrong

`fill_ratio` in `monitoring/core/metrics.py` computes fills divided by orders in.
Those counters are not commensurable. In
`hft_engine/src/auxiliary/order_manager.h`, `orders_in` counts messages in the
NEW state, while `fills` counts every FILLED or PARTIAL_FILL execution report —
and a single crossing trade emits a report for *both* sides against one shard's
`orders_in`. A partially filled order emits several reports before it completes.

The ratio therefore exceeded 1.0 in practice: a fully-filled shard with no
partial fills displayed **"fill%: 200"**.

That has been clamped, so the panel no longer prints an impossible number — but
the clamp only hides the symptom. The displayed value is still not the fraction
of orders that filled, and a plausible-looking wrong number is more dangerous
than an obviously broken one, because it invites trust.

### The fix

Decide what the panel should mean, then measure exactly that. The
straightforward reading — "what fraction of orders that arrived went on to fill"
— cannot be derived from the existing counters and needs a new one in the engine:
a count of orders that reached a terminal filled state, incremented once per
order rather than once per execution report.

That means adding a field to the shared stats region, which carries a hard
constraint: **the region's layout is mirrored by `monitoring/wire.py`.** The
field, the region size, the offsets, and the Python mirror must change in the
same commit, or every reader silently misparses the region. The project's
invariant is that layout changes are versioned — follow it.

If you would rather not extend the engine, the honest alternative is to stop
displaying a ratio and show the raw counters instead, labelled for what they
actually are. Do not keep a clamped ratio that nobody can define.

### Verify

Assert in `monitoring/core/metrics.py`'s self-test that the metric stays within
its defined range for a crossing trade — the case that produced 200%. Then run
the engine with `./build/bin/liquidity`, which produces a deterministic 10,000
orders and 20,000 fills, and confirm the displayed value matches what you decided
it should mean for that known workload.

Both suites must stay green, and the Python stats-region self-test in particular
will fail loudly if the layout and its mirror disagree.

## D5. `generate_pcap` and `replay` cannot produce a single match

**Severity:** medium (a shipped tool cannot do its job) · **Effort:** medium

### What is wrong

Replaying the generated order file produces **zero matches**. A 5,000,000-order
run yields 1,999,996 accepted and **3,000,004 rejected**.

This is *not* the old symbol bug — that is fixed, and the file's symbols decode
correctly. The cause is structural:

- `generate_pcap` writes orders with random sides at prices in a narrow band
  around 50000, which is a design that expects them to cross;
- `replay` opens **one** TCP connection, so every order carries the same
  server-assigned client identity;
- self-trade prevention rejects any order that would cross that same client's
  resting order.

So the flow the generator was designed to create is precisely the flow the engine
must refuse. The pair exercises only the reject path.

This is the same class of problem as two defects the project already fixed: the
symbol mismatch that made the documented benchmark match zero orders, and the
tester's default workload that could never cross. This is the last tool still
carrying it.

For contrast, `liquidity` gets this right — it opens two connections
deliberately, so the resting order at each level always belongs to the other
client and self-trade prevention never fires. Read it before fixing this.

### How to see it

```bash
./build/bin/generate_pcap /tmp/orders.bin
# start the engine, then:
./build/bin/replay /tmp/orders.bin
# decode the audit log; matches will be 0
python3 scripts/decode_audit.py order_audit.log
```

### The fix

Choose one and make the tool honest about it.

**Option 1 — replay over multiple connections.** Split the file across several
connections so crossing orders come from different client identities. This makes
replay produce fills and, as a side benefit, exercises gateway workers properly:
`SO_REUSEPORT` distributes accepted *connections*, so a single-connection client
only ever drives one worker regardless of configuration. Decide how orders map to
connections — an even split is simplest, but note that any two orders on the same
connection still cannot cross each other.

**Option 2 — make the generated flow non-crossing.** Change the generator so the
file is a pure ingest-load fixture, and document it as such. Cheaper, but leaves
the project without a replay path that produces fills.

Option 1 is recommended: it makes the tool do what its name implies, and it is
the only way to replay a workload that exercises matching.

### Verify

Replay the generated file and confirm matches are greater than zero and the
reject rate reflects the workload rather than a structural impossibility. Record
the accepted/rejected split. If you took option 2, verify instead that the
documentation states the file is ingest-only and no doc claims it produces fills.

# Not defects — do not "fix" these

These are known and deliberate. They are listed so nobody files them as bugs.

**Residual market-data loss (~0.8% under burst).** Measured as 240 kernel
`RcvbufErrors` against 29,760 messages applied — the reader itself loses
essentially nothing. This is kernel socket-buffer overflow bounded by
`net.core.rmem_max`, which is a system setting, not a repository one. Raising the
receive buffer beyond the system cap is not possible from application code.

**No gap detection on the ITCH wire.** `tracking_number` exists in the message
struct and is never populated, so consumers cannot detect the loss above. Adding
sequence numbers is a protocol change that was explicitly scoped out. It is the
correct long-term fix for D4 and for the loss above, and it is an *improvement*,
not a defect — but note that the field being declared and never written is
itself misleading to anyone reading the struct.

**End-to-end latency is unmeasured.** Deliberate. Tail latency measures exactly
what arbitrary preemption destroys, and the reference machine cannot grant
`SCHED_FIFO` (`ulimit -r` is 0) and runs a scaling CPU governor. Percentiles
taken there would describe the Linux scheduler, not the engine — the same class
of misleading figure the project already removed once. This needs different
hardware, not a code change.

**Throughput figures are a lower bound.** ~1.08M orders/sec gateway ingest was
measured on a loaded developer desktop; the environment is recorded in the header
of `benchmark_results.txt`. Re-run `scripts/measure_throughput.py` on isolated
hardware to replace it. The scaling *shape* is the meaningful result.

Two notes for whoever picks this up next.

**D3 was resolved differently from the plan below.** The plan proposed adding an
engine-side counter of fully-filled orders. That turned out to be unnecessary:
every match emits exactly two drop copies, so the trade count is exactly half the
`fills` counter, and no stats-region layout change was needed. Removing the
undefined ratio was the honest fix, not redefining it.
